# 기타 코드 운지 교정 웹앱 고도화 바이브코딩 프롬프트

> 목적: 현재 앱에 이미 있는 MediaPipe 손끝 추적, 지판 homography, One-Euro 위치 스무딩, 코드 판정 로직 위에 아직 부족한 세 가지를 실제 동작 코드로 추가한다.
>
> 구현 대상:
> 1. landmark 기반 press/lift 분류기
> 2. 카메라 배치 자동 가이드
> 3. 운지 문법 prior
>
> 기존 시간적 스무딩은 One-Euro 위치 필터와 0.7초 홀드 게이트가 있으므로 삭제하지 않는다. 새로 추가하는 press/lift 결과에도 별도 hysteresis/state smoothing을 붙인다.

---

## 0. 현재 프로젝트 전제

이 프로젝트는 바닐라 HTML/CSS/JS 웹앱이다.

- `index.html`: 비디오, 오버레이 캔버스, 컨트롤 패널
- `style.css`: 다크 테마 UI
- `main.js`: MediaPipe Hand Landmarker, 지판 보정, 손끝 좌표 매핑, 코드 판정, 교정 오버레이
- `README.md`: 실행/사용법

현재 코드에서 유지해야 할 핵심 흐름:

1. `initHandLandmarker()`가 MediaPipe 모델을 로드한다.
2. `startCamera()`가 카메라를 시작한다.
3. `renderLoop()`가 매 프레임 `detectForVideo()`를 호출한다.
4. `mapFingertips()`가 손끝 landmark를 지판 좌표 `(string, fret)`으로 바꾼다.
5. `evaluateVoicing()`가 `detectedFingers`와 `CHORDS`를 비교한다.
6. `drawCorrectionOverlay()`와 `renderJudgePanel()`이 결과를 보여준다.

주의:

- 카메라 영상은 서버 환경에서만 동작한다. `file://` 직접 실행 기준으로 구현하지 않는다.
- 프레임워크나 번들러를 추가하지 않는다.
- 브라우저에서 바로 import 가능한 ES module만 만든다.
- 기존 기능을 깨지 않도록 feature toggle을 둔다.
- 실제 압력 센서가 없으므로 press/lift는 "landmark 기반 접촉 추론"이다. UI 문구도 "추정" 또는 "누름 추정"으로 표현한다.

## 1. 최종 목표

사용자가 기타 지판을 보정하고 목표 코드를 선택하면 앱은 손끝 위치뿐 아니라 다음을 함께 판단한다.

- 이 손가락이 실제로 줄을 누르는 중인지, 단순히 지판 위를 지나가는 중인지, 지판에서 떨어진 상태인지
- 카메라가 너무 멀거나 어둡거나 지판 각도가 심해서 판정 신뢰도가 낮은지
- 현재 손가락 배치가 기타 운지 문법상 가능한지, 또는 바레/손가락 순서/과도한 벌림 규칙에 어긋나는지

최종 `detectedFingers`의 각 항목은 다음 필드를 포함해야 한다.

```js
{
  finger: "1",
  hand: "Left",
  u: 2.4,
  v: 3.1,
  string: 3,
  fret: 2,
  onBoard: true,
  disp: { x: 640, y: 360 },
  posture: { pipAngle: 150, dipAngle: 145, risk: false, message: "" },

  contactState: "pressed",       // "pressed" | "hover" | "lifted" | "unknown"
  contactConfidence: 0.82,       // 0..1
  contactStableMs: 240,
  contactFeatures: {
    cellConfidence: 0.91,
    curlConfidence: 0.72,
    stabilityConfidence: 0.88,
    depthConfidence: 0.45,
    velocityNormPerSec: 0.03,
    reasonCodes: ["on_board", "stable_tip", "curled_finger"]
  },

  prior: {
    score: 0.94,
    violations: [],
    suggestions: []
  }
}
```

## 5. Phase 2: press/lift raw classifier

프롬프트:

```markdown
`src/contact-classifier.js`를 만들어 landmark 기반 raw press/lift/hover 분류기를 구현해줘.

내보낼 함수:

```js
export const CONTACT_STATES = {
  PRESSED: "pressed",
  HOVER: "hover",
  LIFTED: "lifted",
  UNKNOWN: "unknown"
};

export function classifyFingerContact(input, options = {}) {}
export function computeFingerContactFeatures(input, options = {}) {}
export function normalize01(value, min, max, invert = false) {}
```

`classifyFingerContact()` 입력:

```js
{
  finger: "1",
  landmarks,              // MediaPipe hand landmarks array
  joints: { mcp: 5, pip: 6, dip: 7, tip: 8 },
  mapped: {
    onBoard: true,
    string: 2,
    fret: 1,
    u: 1.43,
    v: 4.12
  },
  previousTip: { x: 0.41, y: 0.33, z: -0.03, t: 12.3 },
  currentTip: { x: 0.42, y: 0.34, z: -0.04, t: 12.42 },
  posture: {
    pipAngle: 145,
    dipAngle: 150,
    risk: false
  }
}
```

출력:

```js
{
  state: "pressed",
  confidence: 0.82,
  features: {
    cellConfidence: 0.9,
    curlConfidence: 0.75,
    stabilityConfidence: 0.88,
    depthConfidence: 0.4,
    velocityNormPerSec: 0.05,
    reasonCodes: ["on_board", "near_cell_center", "curled_finger", "stable_tip"]
  }
}
```

판정 규칙:

1. `mapped.onBoard !== true`이면 `lifted`, confidence `0.9` 이상.
2. `mapped.string` 또는 `mapped.fret`이 없으면 `unknown`.
3. `cellConfidence`
   - 손끝이 해당 줄/프렛 셀 중앙에 가까울수록 높다.
   - `u`는 프렛 구간 중앙에 가까울수록 높고, `v`는 줄 인덱스에 가까울수록 높다.
   - 처음에는 단순 휴리스틱으로 충분하다.
4. `curlConfidence`
   - 기타 줄을 누르는 손가락은 완전히 편 손가락보다 PIP/DIP가 어느 정도 굽는다.
   - 기존 `analyzeFingerPosture()`의 `risk`가 true이면 curlConfidence를 낮춘다.
   - 기본 기준:
     - `pipAngle <= 160`이면 긍정
     - `dipAngle <= 165`이면 긍정
     - 둘 다 175도 이상이면 낮음
5. `stabilityConfidence`
   - 손끝 normalized 속도 `velocityNormPerSec`가 낮을수록 높다.
   - `velocityNormPerSec <= 0.08`이면 안정적, `>= 0.30`이면 움직이는 중으로 본다.
6. `depthConfidence`
   - MediaPipe `z`는 카메라/자세에 따라 불안정하므로 약한 가중치만 둔다.
   - tip z와 pip/dip z 차이를 사용하되, 없으면 `0.5`로 둔다.
7. 최종 score:
   - `0.35 * cellConfidence`
   - `0.25 * curlConfidence`
   - `0.25 * stabilityConfidence`
   - `0.15 * depthConfidence`
8. score 기준:
   - `score >= 0.68`: `pressed`
   - `score <= 0.35`: `lifted`
   - 그 외: `hover`

테스트:
- 지판 밖이면 lifted.
- 지판 위이고 손끝이 안정적이며 굽어 있으면 pressed.
- 지판 위이지만 빠르게 움직이면 hover.
- 지판 위이지만 손가락이 완전히 펴져 있고 안정성도 낮으면 hover 또는 lifted. 이 경우 correct로 쓰이지 않으면 된다.
- z가 없어도 예외가 나지 않아야 한다.
```

테스트 예시:

```js
import assert from "node:assert/strict";
import { classifyFingerContact } from "../src/contact-classifier.js";

function lm(x, y, z = 0) {
  return { x, y, z };
}

const joints = { mcp: 5, pip: 6, dip: 7, tip: 8 };

{
  const res = classifyFingerContact({
    finger: "1",
    landmarks: [],
    joints,
    mapped: { onBoard: false, string: null, fret: null, u: -1, v: -1 },
    currentTip: { ...lm(0.5, 0.5), t: 1 }
  });
  assert.equal(res.state, "lifted");
  assert.ok(res.confidence >= 0.8);
}
```

## 6. Phase 3: contact state temporal smoothing

프롬프트:

```markdown
`src/temporal-state.js`를 만들어 raw contact state를 안정화하는 hysteresis state machine을 구현해줘.

내보낼 클래스:

```js
export class ContactStateSmoother {
  constructor(options = {}) {}
  update(raw, tSec) {}
  reset() {}
}

export class ContactSmootherRegistry {
  constructor(options = {}) {}
  update(key, raw, tSec) {}
  markFrame(frameId) {}
  sweepInactive(maxAgeFrames = 2) {}
  clear() {}
}
```

기본 옵션:

```js
{
  pressEnterConfidence: 0.70,
  pressExitConfidence: 0.45,
  hoverEnterConfidence: 0.40,
  minPressMs: 120,
  minLiftMs: 160,
  maxMissingMs: 220
}
```

상태 전이 규칙:

1. 현재 상태가 `lifted` 또는 `unknown`일 때 raw가 `pressed`이고 confidence가 `0.70` 이상으로 120ms 유지되면 `pressed`.
2. 현재 상태가 `pressed`일 때 confidence가 `0.45` 아래로 떨어지거나 raw가 `lifted`로 160ms 유지되면 `lifted`.
3. raw가 `hover`이면 즉시 correct로 쓰지 않는다. UI에는 "눌림 불확실"로 보여준다.
4. 손이 사라져 해당 key가 일정 프레임 갱신되지 않으면 `unknown` 후 registry sweep에서 제거한다.
5. update 결과에는 `state`, `confidence`, `stableMs`, `rawState`, `changed`를 포함한다.

테스트:
- pressed raw가 한 번만 들어오면 아직 pressed가 되지 않는다.
- 120ms 이상 유지되면 pressed가 된다.
- pressed 상태에서 confidence가 잠깐 낮아져도 바로 lifted로 튀지 않는다.
- lifted raw가 160ms 유지되면 lifted가 된다.
```

프롬프트:

```markdown
`main.js`에 contact classifier와 smoother를 통합해줘.

작업:

import { classifyFingerContact } from "./src/contact-classifier.js";

const rawContact = classifyFingerContact({

## 8. Phase 5: evaluateVoicing에 pressed-only 판정 적용

프롬프트:

```markdown
`evaluateVoicing()`를 수정해 contact classifier가 켜져 있으면 pressed 상태만 실제 운지로 인정하게 해줘.

작업:
1. options에 `useContactClassifier`를 추가한다.

```js
function evaluateVoicing(detected, chord, options = {}) {
  const useContactClassifier = !!options.useContactClassifier;
}
```

2. onBoard 필터를 다음처럼 바꾼다.

```js
const onBoard = detected.filter((d) => {
  if (!(d.onBoard && d.string != null && d.fret != null)) return false;
  if (!useContactClassifier) return true;
  return d.contactState === "pressed";
});
```

3. hover 손가락은 별도 배열로 관리한다.

```js
const hovering = detected.filter((d) =>
  useContactClassifier &&

## 10. Phase 7: 운지 문법 prior

프롬프트:

```markdown
`src/fingering-prior.js`를 만들어 운지 문법 prior를 구현해줘.

목표:
- raw landmark 판정만으로 생기는 흔들림을 기타 운지 문법으로 보정한다.
- prior는 정답을 억지로 만드는 장치가 아니다.
- prior는 confidence와 경고/교정 메시지를 제공한다.

내보낼 함수:

```js
export function scoreFingeringPrior(input, options = {}) {}
export function buildTargetMap(chord, capo = 0) {}
export function detectBarreRequirements(chord, capo = 0) {}
export function describePriorViolation(violation) {}
```

`scoreFingeringPrior()` 입력:

```js
{
  chord,
  capo: 0,
  detectedFingers,
  useContactClassifier: true,
  strictFinger: true
}
```

출력:

```js
{
  score: 0.86,
  violations: [
    {
      type: "finger_crossing",
      severity: "warn",
      finger: "1",
      message: "검지가 약지보다 높은 프렛에 있어 운지 순서가 어색합니다."
    }
  ],
  fingerAdjustments: {
    "1": { confidenceMultiplier: 0.8 },
    "3": { confidenceMultiplier: 1.0 }
  },
  suggestions: [
    "검지는 낮은 프렛 쪽, 약지는 높은 프렛 쪽에 두는 것이 자연스럽습니다."
  ]
}
```

규칙:

1. target map
   - `chord.voicing`에서 `fret > 0`인 항목만 목표 운지로 본다.
   - capo가 있으면 기존 `evaluateVoicing()`와 동일하게 fret을 이동한다.
2. open/mute prior
   - target fret이 `0`인 줄은 pressed extras가 있으면 경고.
   - target fret이 `"x"`이고 requireMute가 켜져 있으면 별도 처리.
3. strict finger prior
   - strict mode가 켜져 있으면 목표 손가락과 다른 손가락으로 누른 경우 confidence를 낮춘다.
4. single finger prior
   - 한 손가락이 서로 다른 프렛 두 곳을 동시에 누를 수 없다.
   - 단, 같은 프렛에서 여러 줄을 누르는 것은 바레 후보로 인정한다.
5. barre prior
   - 같은 손가락과 같은 프렛이 2줄 이상 필요하면 바레 requirement로 묶는다.
   - 검지 바레는 자연스럽다.
   - 중지/약지/새끼 바레는 가능하더라도 초보 코드에서는 warn으로 둔다.
6. finger order prior
   - 일반적으로 검지(1)는 중지(2), 약지(3), 새끼(4)보다 같거나 낮은 프렛에 놓이는 경우가 많다.
   - `fingerA < fingerB`인데 `fretA > fretB + 1`이면 crossing warn.
   - 같은 프렛 바레는 예외.
7. stretch prior
   - pressed 손가락들의 fret span이 4를 초과하면 warn.
   - 카포/고프렛 때문에 span 자체가 작으면 경고하지 않는다.
8. adjacency prior
   - detected가 목표 위치에서 string 또는 fret이 1칸 차이이고 같은 손가락이면 "가까운 오검출/이동 필요"로 표시한다.
   - prior가 가까운 위치를 correct로 바꾸지는 않는다.
9. unknown/hover prior
   - hover는 prior score에 약한 감점.
   - unknown이 많으면 "카메라/가림 문제" suggestion을 추가한다.

통합:
1. `evaluateVoicing()` 결과에 `prior` 필드를 추가한다.
2. `renderJudgePanel()`에 prior 경고를 짧게 표시한다.
3. `buildCorrections()`는 prior suggestion 중 severity가 높은 것 1~2개만 추가한다.
4. 새 토글 추가:

```html
<label class="toggle">
  <input type="checkbox" id="fingeringPriorToggle" checked />
  <span>운지 문법 prior</span>
</label>
```

5. 토글이 꺼져 있으면 prior를 적용하지 않는다.

테스트:
- F 코드처럼 검지 같은 프렛 여러 줄은 바레로 인정된다.
- 같은 손가락이 서로 다른 두 프렛을 누르면 violation.
- 검지가 새끼보다 훨씬 높은 프렛에 있으면 crossing warn.
- fret span이 5 이상이면 stretch warn.
- hover가 많으면 score가 낮아지고 camera/press suggestion이 나온다.
```

### 13-1. contact classifier

import assert from "node:assert/strict";
import { classifyFingerContact } from "../src/contact-classifier.js";

const joints = { mcp: 5, pip: 6, dip: 7, tip: 8 };

function makeLandmarks({ tip = {}, pip = {}, dip = {}, mcp = {} } = {}) {
  const arr = Array.from({ length: 21 }, () => ({ x: 0, y: 0, z: 0 }));
  arr[5] = { x: 0.4, y: 0.4, z: 0, ...mcp };
  arr[6] = { x: 0.42, y: 0.42, z: 0, ...pip };
  arr[7] = { x: 0.43, y: 0.45, z: 0, ...dip };
  arr[8] = { x: 0.44, y: 0.47, z: 0, ...tip };
  return arr;
}

{
  const res = classifyFingerContact({
    finger: "1",
    landmarks: makeLandmarks(),
    joints,
    mapped: { onBoard: false, string: null, fret: null, u: -1, v: -1 },
    currentTip: { x: 0.4, y: 0.4, z: 0, t: 1 }
  });
  assert.equal(res.state, "lifted");
}

{
  const res = classifyFingerContact({
    finger: "1",
    landmarks: makeLandmarks(),
    joints,
    mapped: { onBoard: true, string: 2, fret: 1, u: 1.5, v: 4.0 },
    previousTip: { x: 0.40, y: 0.40, z: 0, t: 1 },
    currentTip: { x: 0.405, y: 0.405, z: 0, t: 1.2 },
    posture: { pipAngle: 145, dipAngle: 150, risk: false }
  });
  assert.equal(res.state, "pressed");
  assert.ok(res.confidence >= 0.68);
}
```

## 17. 구현자가 피해야 할 실수

- `pressPoint()`를 press/lift classifier로 착각하지 마라. 이것은 목표 지점 표시용 좌표 함수다.
- MediaPipe z값만으로 "눌림"을 단정하지 마라. z는 약한 보조 feature로만 써라.
- hover를 correct로 처리하지 마라.
- prior가 raw detection을 무조건 덮어쓰게 만들지 마라. prior는 score와 warning을 제공하고, 최종 판정은 pressed + 위치 + prior를 함께 본다.
- 카메라 가이드가 poor라고 판정을 완전히 막지 마라. 신뢰도 낮음 경고를 보여주는 것이 우선이다.
- UI에 긴 설명을 넣지 마라. 사용자가 바로 행동할 수 있는 짧은 지시문으로 표시하라.
- 기존 One-Euro 위치 스무딩을 제거하지 마라.
- 기존 지판 보정 저장/localStorage 동작을 깨지 마라.
